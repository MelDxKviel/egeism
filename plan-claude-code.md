# ЕГЭ-платформа — план реализации для Claude Code

Спецификация проекта для агентной реализации. Организована так, чтобы часть
работы шла параллельно (subagents / git worktrees), но при этом агенты не
конфликтовали по коду.

---

## 0. Как пользоваться этим документом

1. Раздел **1–4** (обзор, стек, архитектура, контракты) — скорми Claude Code как
   первичный контекст. Раздел **3–4** дистиллируй в `CLAUDE.md` в корне репо:
   агент должен видеть конвенции и архитектуру в каждой сессии.
2. Раздел **5 (Phase 0)** делается **строго последовательно, в одной сессии, и
   лочится человеком**. Это фундамент — общие типы, схема БД, контракт API.
   Пока он не заморожен, параллелить нельзя (иначе merge-ад по схеме).
3. После заморозки контрактов — раздел **6 (workstreams)** можно вести
   параллельно. Границы владения по пакетам расписаны так, чтобы агенты не
   лезли в одни файлы.
4. Раздел **7 (checker)** — сердце проекта. Спека точная специально: по ней
   агент строит движок сравнения ответов, а ты потом проверяешь его на реальных
   ответах ЕГЭ. Это единственное место, где я настаиваю на ручной верификации.
5. Раздел **10** — как именно оркестрировать это в Claude Code (subagents,
   worktrees, opusplan).

## 1. Что строим

Платформа подготовки к ЕГЭ: **сайт + Telegram-бот + админка**, общий backend-API.

**Scope первого этапа (жёстко):**
- Только **первая часть** каждого предмета — задания, где ответ это конечное
  значение, сравнимое автоматически (число / слово / последовательность / множество).
- Четыре предмета: русский, математика, информатика, обществознание.
- Пользователи: **1 ученик + 1 учитель**, но модель данных сразу multi-user.

**Вне scope первого этапа** (не строим, но не ломаем расширяемость):
- Проверка кода информатики в sandbox.
- LLM-проверка сочинений/развёрнутых ответов по критериям.
- Группы/классы, оргструктура, реальный auth-провайдер.

## 2. Стек (решения приняты — не пересогласовывать)

| Слой | Технология | Обоснование |
|---|---|---|
| Backend API | **Go** + `chi` (router) | единый источник бизнес-логики; стек уже знаком |
| БД | **PostgreSQL** + JSONB | `answer_schema` разных форм в JSONB |
| Доступ к БД | **sqlc** + **pgx** | type-safe SQL без ручного boilerplate |
| Миграции | **goose** | простые up/down миграции |
| Telegram-бот | **Go** (`telego` или та либа, что в servmon-bot) | консистентность с текущим проектом |
| Очередь/шедулер | **asynq** (поверх Redis) | нативные delayed/scheduled задачи — ровно под уведомления о тестах |
| Object storage | **MinIO** (S3 API) | медиа заданий (картинки, файлы) |
| Frontend | **React** (Vite) + TanStack Query | UI приходит из Claude Design, провязывается на этот же API |
| Графики | Recharts (mastery) + кастом/либа для heatmap | |
| Деплой | **docker-compose** | одна семья — k8s избыточен; переезд на k8s возможен позже |

## 3. Архитектура

```
                    ┌──────────────┐
   Web (React) ───► │              │
                    │   Go API     │ ──► PostgreSQL
   Bot (Go)   ────► │  (вся логика │ ──► MinIO (медиа)
                    │  + checker)  │ ──► Redis ◄── asynq worker
   Admin UI ──────► │              │        │
                    └──────────────┘   scheduled notifications ─► Telegram
```

**Железные правила:**
- Вся бизнес-логика (в первую очередь **проверка ответов**) живёт в API-слое и
  **не дублируется в боте**. Бот — тонкий клиент.
- Бот и веб ходят в **один и тот же** набор эндпоинтов. Разница только в UI и в
  том, какие типы заданий «bot-solvable» (см. §8).
- Авторизация бота — по `telegram_id` → маппинг на `users`. Никаких паролей на
  первом этапе.

**Раскладка репо (границы владения = границы пакетов):**

```
/cmd
  /api          main API-сервера
  /bot          main бота
  /worker       main asynq-воркера (шедулер + обработчики)
  /ingest       CLI ингеста контента
/internal
  /domain       core-типы, сущности         [Phase 0 — orchestrator]
  /store        sqlc-код + запросы к БД      [Phase 0 схема, далее per-feature]
  /checker      движок сравнения ответов     [WS-A, изолирован]
  /api          HTTP-хендлеры, роутинг        [WS-A / WS-C]
  /bot          хендлеры бота                 [WS-B]
  /scheduler    asynq-задачи и хендлеры       [WS-D]
  /stats        агрегации статистики          [WS-C]
  /ingest       адаптеры источников контента  [WS-E, полностью изолирован]
  /scoring      прогноз балла                 [позже]
/migrations     goose-миграции               [Phase 0 + per-feature]
/deploy         docker-compose, Dockerfiles   [WS-F]
/web            React-приложение (из Design)  [интеграция]
/api/openapi.yaml  контракт API              [Phase 0 — заморозить]
```

**Шов, который держит параллельность:** все зависят от `internal/domain` и
интерфейсов `internal/store`. Phase 0 их замораживает → дальше пакеты пишутся
независимо.

## 4. Модель данных

```sql
users         id, role(student|teacher), telegram_id, name, created_at
enrollments   id, teacher_id→users, student_id→users   -- m2m готов, 1 строка сейчас
subjects      id, code(rus|math|inf|soc), title

tasks         id, subject_id, number(номер в ЕГЭ), statement,
              media(jsonb),           -- ссылки на объекты в MinIO
              answer_schema(jsonb),   -- см. §7
              source(jsonb),          -- откуда пришло (для ингеста/дедупа)
              status(draft|active|rejected)  -- курация авто-ингеста
              created_at

tests         id, subject_id, kind(classic|drill), title, created_by→users, created_at
test_items    id, test_id, task_id, position

assignments   id, test_id, student_id→users, assigned_by→users,
              scheduled_at, notified_at(nullable), status(scheduled|done|missed)
attempts      id, assignment_id(nullable), test_id, student_id→users,
              started_at, finished_at(nullable)
answers       id, attempt_id, task_id, raw_answer, is_correct,
              time_spent_ms, answered_at
```

**Ключевые решения модели:**
- `assignments` (учитель назначил) и `attempts` (ученик прошёл) — **разведены**.
  `assignment_id` в `attempts` nullable → ученик может решать и сам, без назначения.
- `answers.is_correct` + `time_spent_ms` кормит **всё**: heatmap (агрегат по дням),
  слабые места (group by `tasks.number`, средний %), время на задание.
- Отдельные таблицы статистики на двух юзерах не нужны. Упрёшься в тормоза —
  materialized view. Не раньше.
- `tasks.status` — потому что авто-ингест не идеален; плохие задания уходят в
  `draft/rejected`, а не в бой.

## 5. Phase 0 — фундамент (ПОСЛЕДОВАТЕЛЬНО, лочится человеком)

Цель: заморозить контракты, на которые дальше опираются все параллельные потоки.

- [ ] Скелет репо, `go.mod`, раскладка пакетов из §3.
- [ ] `docker-compose` с postgres + redis + minio, всё поднимается локально.
- [ ] goose-миграции: полная схема из §4.
- [ ] sqlc настроен, генерит типы из схемы, компилируется.
- [ ] `internal/domain`: все core-типы + enum'ы. Здесь же — Go-представление
      `answer_schema` (см. §7) как discriminated union по `type`.
- [ ] `internal/store`: интерфейсы репозиториев (Users, Tasks, Tests,
      Assignments, Attempts, Answers) + sqlc-реализация базовых CRUD.
- [ ] `api/openapi.yaml`: контракт всех эндпоинтов (даже ненаписанных) — это то,
      на что провяжется фронт из Claude Design.
- [ ] `/health` эндпоинт отвечает, API стартует.

**🔒 Human checkpoint:** ты ревьюишь и замораживаешь схему БД + `openapi.yaml` +
типы `answer_schema`. Пока не залочено — параллельные потоки не стартуют.

## 6. Workstreams (параллелятся после Phase 0)

Каждый WS = отдельный subagent или worktree. Владение файлами не пересекается.

### WS-A — Ядро API + checker
Владеет: `internal/checker`, `internal/api` (кроме admin), `cmd/api`.
- [ ] Реализовать движок сравнения ответов по §7 **с полным табличным
      тест-сьютом** (это safety net).
- [ ] Эндпоинты ученика: получить задачу/тест, отправить ответ (→ checker →
      записать `answer`), получить свою статистику (heatmap, per-task).
- [ ] Старт/финиш `attempt`, учёт `time_spent_ms`.
- **Done:** ученик через API решает задачу, ответ корректно проверяется и
  пишется, статистика отдаётся.

### WS-B — Telegram-бот
Владеет: `internal/bot`, `cmd/bot`.
- [ ] Регистрация по `telegram_id`, deep-link `t.me/bot?start=<token>` для
      привязки и хендоффа на веб.
- [ ] Решение bot-solvable заданий прямо в чате (см. §8), запись через API.
- [ ] Приём и показ уведомления о запланированном тесте.
- [ ] Разбор после неверного ответа (не просто «неверно»).
- **Done:** ученик решает короткие задания в боте, результат в общей статистике.

### WS-C — Админ-API + статистика
Владеет: `internal/stats`, `internal/api/admin`.
- [ ] Эндпоинты статистики по ученику: несколько предметов, per-task график
      успешности, слабые места (топ номеров с худшим %), время на задание.
- [ ] Drill-down: клик по клетке heatmap → какие задачи/варианты решались в день.
- [ ] Браузер/редактор банка задач (см. §9 — **курация авто-ингеста**, не ручной
      ввод): список, фильтры по предмету/номеру, правка ответа, `active/reject`.
- [ ] Конструктор тестов: `classic` (как на ЕГЭ) и `drill` (N задач одного номера).
- [ ] Экспорт теста в PDF, стилизованный под бланк с полем для ответа.
- **Done:** учитель видит подробную статистику, собирает и экспортирует тесты.

### WS-D — Шедулер назначений
Владеет: `internal/scheduler`, `cmd/worker`.
- [ ] asynq-задача «уведомить о тесте в `scheduled_at`» — enqueue при создании
      assignment, отправка в Telegram по времени, простановка `notified_at`.
- [ ] Нуджи по стрику («не решал 3 дня») как периодическая задача.
- **Done:** назначил тест на время → в срок прилетает уведомление в ТГ.

### WS-E — Ингест контента (изолирован, рисковый — см. §9)
Владеет: `internal/ingest`, `cmd/ingest`.
- [ ] **Сначала — задача 0: оценить источники** (§9) и выбрать. Это решение за
      человеком, влияет на качество.
- [ ] Адаптер(ы) источника → нормализация → запись в `tasks` со `status=draft`.
- [ ] Дедуп по `source`. Заливка медиа в MinIO.
- **Done:** банк задач наполняется автоматически, задачи уходят в админку на курацию.

### WS-F — Инфраструктура
Владеет: `deploy/`, Dockerfile'ы, Makefile, CI.
- [ ] Dockerfile'ы на api / bot / worker / ingest, multi-stage.
- [ ] docker-compose прод-профиль: api, bot, worker, postgres, redis, minio, web (nginx).
- [ ] Прогон миграций как init-шаг. Bucket MinIO создаётся при старте.
- [ ] Makefile: `make dev`, `make migrate`, `make test`, `make build`.
- [ ] (Опц.) GitHub Actions: сборка + тесты checker'а.
- **Done:** `docker-compose up` поднимает весь стек с нуля.

## 7. Движок сравнения ответов (СЕРДЦЕ — спека точная)

`answer_schema` в `tasks` — JSONB, дискриминируется полем `type`. Не «строка
правильного ответа», а описание того, **как** сравнивать. Наивное
`ответ == правильный` даст ложные «неверно» → ученик бросит через день.

### Типы

**`number`** — одно числовое значение.
```json
{"type":"number","correct":["0,5"],"tolerance":0.001}
```
Нормализация обеих сторон: `,`→`.`; strip пробелов; unicode-минус (U+2212,
U+2013, U+2014)→`-`; парсить как decimal; сравнивать с `tolerance` (abs).
`correct` — массив (может быть несколько допустимых форм). **Важно:** коды вроде
«35» или «24», где это цифры 3-5, а не число тридцать пять — это **`sequence`**,
не `number`.

**`string`** — слово/фраза.
```json
{"type":"string","correct":["ещё","еще"],"ci":true,"yo_fold":true}
```
trim; collapse внутренних пробелов; `ci`→lowercase обеих сторон; `yo_fold`→
`ё`→`е` обеих сторон; сравнить с любым из `correct`.

**`set`** — множество, порядок НЕ важен («укажите номера»).
```json
{"type":"set","correct":["2","4","5"],"token":"char"}
```
Токенизация по `token`: `"char"` — убрать разделители, каждый символ = токен
(«245» = «2,4,5» = «2 4 5»); `"split"` — резать по разделителям (для
многосимвольных токенов). Сравнить как множества (dedup, порядок игнор).

**`sequence`** — упорядоченная последовательность (обществознание, многие коды).
Как `set`, но сравнение как упорядоченный список.

### Тест-сьют checker'а — ОБЯЗАН покрывать (это safety net)

| Кейс | Пример |
|---|---|
| Запятая vs точка | `0,5` == `0.5` |
| Unicode-минус | `−3` (U+2212) == `-3` |
| Пробелы/reg в строке | ` Ещё ` == `еще` при ci+yo_fold |
| ё/е фолдинг | `ещё` == `еще` |
| set без разделителей | `245` == `2,4,5` == `524` |
| sequence — порядок важен | `312` != `123` |
| множество допустимых | ответ матчит любой из `correct` |
| tolerance | `3.14159` == `3.14` при нужном допуске |
| негатив | пустой ответ, мусор, лишние символы → `false` |

**🔒 Human checkpoint:** перед тем как доверять checker'у — прогони его вручную
на ~20–30 реальных ответах части 1 **по каждому предмету** (демоверсии ФИПИ).
Молчаливый баг здесь = продукт засчитывает правильное как неправильное = провал.
Это единственное место, где я категорически прошу ручную проверку.

## 8. Граница бот/веб

В чате нормально решаются только короткие ответы. Помечай в `tasks` (или выводи
из `answer_schema` + наличия медиа) флаг `bot_solvable`:
- **bot-solvable:** number / string / короткие set-sequence без картинок.
- **только веб:** задания с несколькими картинками, таблицами, файлами
  (пригодится, когда дойдёшь до информатики с файлами).

Deep-link хендофф: в боте кнопка «продолжить на сайте» → `t.me/bot?start=<token>`
→ веб открывает ту же сессию решения по токену.

## 9. Ингест контента — честно про «без рук»

Это единственное место, где «полностью автоматом» конфликтует с «правильные
ответы». Разложил варианты, решение за тобой (влияет на качество):

- **open.fipi.ru (открытый банк ФИПИ)** — единственный **легальный** первоисточник
  настоящих заданий. Минусы: отдаёт задания **картинками** с вшитыми формулами;
  **готовых ответов для многих заданий нет** (их надо решать). Для авто-проверки
  без ответов бесполезен без доп. слоя.
- **sdamgia / РЕШУ ЕГЭ** — есть **и задания, и ответы**, структурировано; есть
  неофициальная либа-обёртка. Минус: это **против их ToS**, есть анти-скрейпинг.
  Технически работает, юридически серая зона. Решать тебе — фиксирую факты без
  морали.
- **Готовые датасеты (HuggingFace/Kaggle)** — если найдётся приличный датасет
  ЕГЭ с ответами, это самый чистый «без рук + есть ответы». Проверить наличие —
  первая задача WS-E.
- **LLM-ассист** (OCR картинок ФИПИ + решение моделью) — закрывает «без ответов»,
  но ответы LLM для ЕГЭ (особенно математика) **ненадёжны**. Только с обязательной
  курацией в админке, не в бой напрямую.

**Архитектурный вывод:** ингест — **изолированный адаптер**, чей единственный
контракт — писать строки в `tasks` (со `status=draft` и заполненным
`answer_schema`). Остальной системе всё равно, откуда данные. Меняешь источник —
меняешь один пакет. Поэтому WS-E можно начать позже всех и не блокировать им
остальное.

## 10. Оркестрация в Claude Code (параллельность на практике)

Главный трезвый тезис: **бутылочное горлышко — не вычисления агентов, а твоё
ревью.** Поэтому: параллель мельком там, где механика и низкий риск; сериализуй
там, где последствия.

**Что параллелить (изолировано, есть safety net или дёшево ретрофитить):**
`internal/checker` (его тест-сьют — сеть безопасности), `internal/ingest`,
`internal/scheduler`, `deploy/`. → отдельные **subagents** внутри сессии или
отдельные **worktrees**.

**Что сериализовать (высокий риск, требует твоего ревью):** схема БД и миграции,
`openapi.yaml`, корректность checker'а, всё что трогает реальные данные ученика.
Изменения схемы **обязаны идти через оркестратор-сессию по одному** — два агента,
одновременно правящих миграции, гарантируют конфликт.

**Практический рецепт:**
1. `opusplan`: разверни каждый workstream (§6) в список конкретных задач.
2. **Phase 0** — одна сессия, последовательно, залочил контракты.
3. Для изолированных пакетов (checker, ingest, scheduler, infra) — заведи
   **кастомные subagents** по одному на пакет, гоняй в рамках сессии.
4. Для по-настоящему независимых бинарей (**бот**, интеграция **web**) — отдельный
   **git worktree** + отдельный терминал, если хочешь выигрыш по wall-clock.
5. API/store держи на `main` в оркестраторе, чтобы не ловить конфликты по схеме.
6. `/advisor` — на спорных архитектурных развилках.

## 11. Вехи (каждая — вертикальный срез, проверяемый руками)

- **M0 (Phase 0):** стек поднимается, схема мигрирована, домен+sqlc, `/health`,
  черновик openapi. → **человек лочит контракты.**
- **M1 (тонкий срез):** учитель добавил одну задачу (`number`) → ученик решил в
  вебе (мок-UI ок) → checker проверил верно → `attempt`+`answer` записаны →
  видно в базовой статистике. **Доказывает хребет.**
- **M2:** все 4 типа в checker'е (+ тест-сьют), бот решает задание, эндпоинты
  heatmap и per-task mastery.
- **M3:** конструктор тестов (classic+drill), assignments, asynq-уведомление в
  ТГ по времени срабатывает, PDF-экспорт.
- **M4:** пайплайн ингеста наполняет реальный банк задач.
- **M5 (киллер-фича):** прогноз балла — первичный балл по % решаемости → тестовый
  по официальной шкале перевода. Нуджи по стрику.

## 12. Definition of Done проекта (этап 1)

- `docker-compose up` с нуля поднимает рабочую систему.
- Учитель: собирает тесты из авто-наполненного (и отревьюенного) банка,
  назначает ученику на время с уведомлением в ТГ, экспортирует в PDF, видит
  подробную статистику по предметам и номерам задач.
- Ученик: решает в вебе и в боте, видит heatmap со стриком, per-task прогресс,
  прогноз балла, разбор после ошибок.
- Checker верифицирован вручную на реальных ответах всех 4 предметов.
- Модель данных multi-user, добавление ещё людей не требует переписывания.
