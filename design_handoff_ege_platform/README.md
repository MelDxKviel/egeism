# Хендофф: ЕГЭ-платформа «Вектор» — фронтенд

## Overview
Платформа подготовки к ЕГЭ для семьи: **ученик** решает задания части 1 (сайт + Telegram-бот),
**учитель** собирает тесты, назначает их и следит за прогрессом по каждому номеру задания.
Этот пакет — дизайн всего веб-фронтенда (11 экранов: флоу ученика + админка учителя),
собранный на мок-данных. Задача Claude Code: **воссоздать эти экраны в реальном React-приложении
и провязать на backend по контракту `openapi.yaml`**.

## About the Design Files
Файл `EGE-Platform.dc.html` — это **дизайн-референс, а не продакшн-код**. Это интерактивный
HTML-прототип, показывающий итоговый вид, тексты, состояния и поведение. Его нужно **пересобрать
в целевом стеке (React)** по устоявшимся паттернам проекта (или, если проекта ещё нет — выбрать
стек: React + Vite + TypeScript, стейт через React Query для серверных данных + локальный стейт
экранов, графики — Recharts или lightweight SVG как в прототипе). **Не копировать HTML напрямую.**

Прототип написан как один файл со стейт-машиной и клиентскими мок-данными в форме будущего API.
Пиксельная истина (отступы, размеры, цвета конкретных элементов) — в самом файле; открой его в
браузере и инспектируй. Ниже — система (токены, лейаут, экраны, состояния) и карта провязки на API.

## Fidelity
**High-fidelity.** Финальные цвета, типографика, отступы, интеракции, копирайт — воссоздавать
пиксель-в-пиксель средствами вашей дизайн-системы/библиотек.

## Экраны / Views
Ниже — назначение, структура и провязка. Точные размеры/копирайт — в `EGE-Platform.dc.html`.
`{sid}` = id ученика, `{subject}` = код предмета (`rus|math|inf|soc`).

### Ученик
1. **Дашборд** (`dashboard`). Сетка 12-колонок, на мобиле — 1 колонка.
   - *Прогноз балла* (крупное число `72 / 100`, чип тренда, «N из M первичных», точность, мотив-подпись).
     → `GET /api/students/{sid}/stats/forecast?subject={subject}` (схема `Forecast`).
   - *Слабые места* (2–4 худших номера, бар точности, кнопка «Прокачать» → drill).
     → `GET /api/students/{sid}/stats/weak-spots?subject={subject}`.
   - *Активность* (heatmap-год по дням, стрик «N дней подряд», клик по клетке → history).
     → `GET /api/students/{sid}/stats/heatmap?since=YYYY-MM-DD`. Стрик считается на клиенте.
   - *Назначено тебе* (карточки тестов с временем/статусом, кнопка «Начать»).
     → нужен `GET` списка назначений (см. «Пробелы контракта»).
   - Состояния: **пусто** (онбординг), **загрузка** (скелетоны), **ошибка** (ретрай).
2. **Экран предмета** (`subject`). Переключатель 4 предметов + сетка номеров (спарклайн, %, попытки).
   Клик по номеру → модалка с **графиком mastery во времени** + «Решать N таких» (drill).
   → `GET /api/subjects` (переключатель), `GET /api/students/{sid}/stats/mastery?subject={subject}`.
   Для линии во времени нужен ряд точек (см. «Пробелы»).
3. **Решение теста** (`solve`) — ядро UX. Одна задача/экран: прогресс `3 / 15`, **таймер** (обратный отсчёт),
   тело задания, плейсхолдер медиа (image/table/file), навигация по номерам (дотсы), «Ответить».
   - **AnswerInput адаптируется под `answer_kind`:** `number` → числовой ввод; `string` → текст;
     `sequence` → экранная цифровая клавиатура + чипы; `set` → числа через пробел + превью-чипы.
   → старт: `POST /api/practice {subject}` (свободно) или `POST /api/attempts {test_id, assignment_id}` (назначенный);
   загрузка задачи: `GET /api/tasks/{taskID}`; ответ: `POST /api/attempts/{attemptID}/answers` (см. §3.4).
4. **Разбор после ответа** (в том же экране). При ошибке — красный вердикт «Пока неверно» + **прогрессивные
   подсказки** (раскрываются по одной), затем полный «Разбор решения». `solution[]` приходит только при неверном.
   → из ответа `POST …/answers` → `SubmitAnswerResponse.solution`.
5. **Итоги теста** (`results`). Кольцо процента, верно/всего, время, дельта к прогнозу, разбивка по задачам.
   → `POST /api/attempts/{attemptID}/finish` + `GET /api/attempts/{attemptID}/answers` для разбивки.
6. **История** (`history`). Большой heatmap + «Недавние решения». Клик по дню → drill-down списка задач дня.
   → `GET /api/students/{sid}/stats/day?date=YYYY-MM-DD`. Лента решений — см. «Пробелы».

### Учитель (админка)
7. **Дашборд учителя** (`t-dashboard`). Карточка ученика (прогноз, стрик, точность, назначено), свежие
   попытки, слабые места с «Дрилл». → те же stats-эндпоинты для конкретного `{sid}`.
8. **Статистика ученика** (`t-student`). Переключатель предметов; **бар-чарт успешности по каждому номеру**
   (цвет по точности); слабые места; «Дольше всего решает» (по `avg_time_ms`).
   → `mastery` + `weak-spots`. Время на задание берётся из `NumberMastery.avg_time_ms`.
9. **Конструктор тестов** (`t-builder`). Тип **classic** (структура ЕГЭ) / **drill** (N задач одного номера),
   выбор задач из банка (только `active`), предпросмотр, сохранение.
   → `POST /api/admin/tests {kind}` + `POST /api/admin/tests/{testID}/items` на каждую задачу.
10. **Назначение** (`t-assign`). Тест → ученик → время (`datetime-local`) → тумблер «уведомить в Telegram».
    Альтернатива: «Экспорт в PDF-бланк». → `POST /api/admin/assignments` (кладёт задачу на TG-уведомление).
11. **Банк задач** (`t-bank`) — курация авто-ингеста. Фильтры по статусу/номеру, карточки задач
    (тело + медиа + тип ответа + флаг «бот решает»), **правка правильного ответа**, «Одобрить/Отклонить».
    → список: `GET /api/tasks?status=&subject=&number=`; правка ответа: `PATCH /api/admin/tasks/{taskID}/answer`
    (тело — схема `AnswerSchema`); статус: `PATCH /api/admin/tasks/{taskID}/status`.

## Interactions & Behavior
- Навигация — переключение вида (стейт), без перезагрузки. Роль (ученик/учитель) и тема — глобальный стейт, персист в localStorage.
- Таймер теста — обратный отсчёт от лимита; по нулю — авто-финиш. Позицию/таймер стоит персистить, чтобы решение переживало перезагрузку.
- Прогрессивные подсказки: раскрываются по одной (не весь ответ сразу); полный разбор — после подсказок.
- Анимации входа секций — короткий fade/slide (~0.3s ease). Ховеры на кнопках/карточках/навигации.
- Каждый экран имеет **пусто / загрузка (скелетоны) / ошибка**. На дашборде есть демо-переключатель состояний (в проде убрать — состояния диктуются реальными запросами).
- Адаптив обязателен: дашборд и экран решения удобны на телефоне (ученик заходит после хендоффа из бота).

## State Management
- Серверные данные (subjects, tasks, stats, attempts) — рекомендуется React Query: кэш, загрузка/ошибка из коробки.
- Локальный стейт экрана решения: `attempt {test_id, task_ids[], idx, answers{}, secondsLeft}`, черновик ответа, вердикт, шаг подсказки.
- Глобально: роль, тема, выбранный предмет.
- Auth (этап 1): заголовок `X-User-ID` на всех запросах (кроме `security: []`). Реальный вход ученика — `POST /api/auth/telegram` (deep-link хендофф из бота).

## Проверка ответа (§7)
Сравнение — **на сервере** (`POST …/answers`). Клиентский чекер в прототипе только для мока и повторяет `AnswerSchema`:
`number` (абсолютный `tolerance`), `string` (`ci`, `yo_fold` ё→е), `set`/`sequence` (`token: char|split`; set — без учёта порядка, sequence — с порядком). Не тащить проверку на клиент в проде.

## Пробелы контракта (UI требует, в openapi пока нет)
Прототип использует данные, для которых стоит добавить эндпоинты:
1. **Список назначений ученику** (дашборд «Назначено тебе», учитель) — напр. `GET /api/students/{sid}/assignments` или `GET /api/assignments?student=`.
2. **Лента попыток** (ученик «Недавние решения», учитель «Свежие попытки») — напр. `GET /api/attempts?student=&limit=`.
3. **Динамика mastery во времени** (график номера + спарклайны в сетке): сейчас `/stats/mastery` отдаёт агрегат (`total/correct/avg_time_ms`). Нужен ряд точек во времени — напр. `?number=&series=weekly` или `/stats/mastery-series`.
4. **Список/детали тестов** для конструктора и назначения — есть только `POST /api/admin/tests`; нужен `GET`.
5. **Экспорт PDF-бланка** назначенного варианта (кнопка в «Назначении») — в прототипе заглушка.

До появления этих эндпоинтов их можно держать на мок-слое (формы данных — как в прототипе).
