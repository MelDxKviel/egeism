# ЕГЭ-платформа — бриф для Claude Design (фронтенд)

Задача Claude Design: спроектировать и собрать **фронтенд сайта** (React). Только
UI и клиентская логика на **мок-данных**. Провязка на реальный backend-API
делается позже в Claude Code по контракту `api/openapi.yaml`.

---

## 1. Продукт в одну строку

Платформа подготовки к ЕГЭ для семьи: ученик решает задания части 1 (сайт + бот),
учитель собирает тесты, назначает их и следит за прогрессом по каждому номеру задания.

**Пользователи:** один ученик, один учитель (роли уже сейчас разведены; UI
рассчитан на рост до нескольких человек).
**Язык интерфейса:** русский.
**Устройства:** сайт адаптивный. Ученик может заходить с телефона (после хендоффа
из Telegram-бота), учитель чаще с десктопа. Оба сценария — first-class.

---

## 2. Тон и визуальный язык

- Настроение: **спокойно, сфокусированно, мотивирующе.** Не корпоративный SaaS,
  не детский. Инструмент, к которому возвращаются каждый день без раздражения.
- Чистая типографика, воздух, мягкие акценты. Данные (heatmap, графики) —
  читаемы и приятны, это визитная карточка продукта.
- Тёмная и светлая темы желательны (ученик может решать вечером).
- Единый набор дизайн-токенов (цвет / типографика / отступы / радиусы),
  переиспользуемый по всем экранам.

**Сигнатурные элементы, на которые сделать акцент:**
1. **Heatmap активности** «как в гитхабе» — клетки по дням, интенсивность по
   активности, клик по клетке → что решалось в этот день.
2. **График mastery по номеру задания** — для каждого номера предмета видно,
   насколько успешно он сейчас решается (динамика во времени).
3. **Прогноз балла** — крупный, заметный: «сейчас ты примерно на 72».
   Это главный мотиватор и то, что отличает от обычной дрочильни задач.

## 3. Экраны — УЧЕНИК

### 3.1 Главная (dashboard)
- **Прогноз балла** по выбранному предмету — крупный gauge/число + короткий
  подпись-контекст.
- **Heatmap активности** со стриком («N дней подряд»).
- **Назначено тебе** — карточки тестов, которые учитель запланировал (с временем).
- **Слабые места** — 2–3 номера задач с худшим %, кнопка «прокачать» (→ drill).
- Быстрый вход: «решать» (выбор предмета).
- Состояния: пустое (ничего не решал — приветственный онбординг), загрузка, ошибка.

### 3.2 Экран предмета
- Переключатель предметов: русский / математика / информатика / обществознание.
- Сетка/список **номеров задания** предмета, у каждого — индикатор текущей
  успешности (%, мини-спарклайн).
- Клик по номеру → **график mastery** этого номера (динамика во времени) +
  кнопка «решить N таких» (drill).

### 3.3 Решение теста (ядро UX)
- Одна задача на экран. Прогресс «3 / 15», **таймер** (экзамен на время).
- Тело задания: текст + возможные картинки/таблицы.
- **Поле ответа адаптируется под тип** (это важно):
  - число — числовой ввод;
  - слово — текстовый;
  - последовательность/множество — ввод последовательности цифр (или
    выбор/перетаскивание, если так удобнее).
- Кнопки: «ответить», навигация между задачами, «завершить».
- Состояния: не отвечено / отвечено / текущее.

### 3.4 Разбор после ответа
- Не просто «верно/неверно» — при ошибке сразу **разбор** (цель научить, а не
  оценить).
- Прогрессивные подсказки: раскрываются по шагам, а не сразу весь ответ.

### 3.5 Итоги теста
- Сколько верно, время, разбивка по номерам.
- Как это двигает прогноз балла.

### 3.6 История (drill-down)
- Открывается по клику на клетку heatmap: список задач/вариантов, решённых в тот
  день, с результатом.

## 4. Экраны — УЧИТЕЛЬ (админка)

### 4.1 Дашборд учителя
- Обзор по ученику(-ам): активность, прогноз балла, что назначено, свежие попытки.

### 4.2 Детальная статистика по ученику
- Несколько предметов в одном месте.
- По каждому предмету — **график успешности по каждому номеру задания**.
- **Слабые места** (топ худших номеров), **время на задание**, тренды.

### 4.3 Конструктор тестов
Два вида на предмет:
- **Классический вариант** — как на ЕГЭ (набор номеров по структуре экзамена).
- **Drill** — N задач одного конкретного номера.
- Выбор задач из **банка** (§4.5), предпросмотр варианта.

### 4.4 Назначение теста
- Выбрать тест → выбрать ученика → **запланировать время** → тумблер
  «уведомить в Telegram».
- Альтернатива: **экспорт в PDF** (стилизован под реальный бланк с полем для
  ответа — чтобы репетиция была приближенной).

### 4.5 Банк задач (курация авто-ингеста)
> Важно по контексту: задания наполняются **автоматически**, пользователь их
> руками не вводит. Но авто-ингест не идеален, поэтому админке нужен **браузер и
> редактор** для ревью, а не форма ручного создания.
- Список задач, фильтры по предмету/номеру/статусу.
- Просмотр задания и его правильного ответа.
- Правка ответа, пометка `active` / `reject` (черновики из ингеста не попадают в
  бой, пока не одобрены).

## 5. Библиотека компонентов (переиспользуемые)

- **HeatmapГрид** — год активности по дням, кликабельные клетки, tooltip.
- **MasteryChart** — линейный график успешности номера во времени (Recharts-friendly).
- **ScoreGauge** — прогноз балла, крупно.
- **TaskCard** — карточка задания (тело + медиа + поле ответа).
- **AnswerInput** — варианты под тип ответа (число / слово / последовательность).
- **Timer** — обратный отсчёт для теста.
- **WeakSpotsList** — слабые номера с CTA «прокачать».
- **AssignmentCard** — назначенный тест с временем и статусом.
- **SubjectSwitcher** — переключатель 4 предметов.
- **TestBuilder** — сборка classic/drill.
- **StatBlock/эмпти-стейты** — единый паттерн загрузки/пустого/ошибки везде.

## 6. Данные и состояния (для мок-слоя)

Сделай моки, повторяющие формы будущего API (потом Claude Code провяжет на
реальные эндпоинты по `openapi.yaml`):
- список предметов; задачи с типом ответа и медиа; тест (набор задач);
  назначения (с временем/статусом); попытки и ответы (верно/время);
  агрегаты для heatmap (день → активность) и mastery (номер → точки во времени);
  прогноз балла (первичный → тестовый).

Каждый экран обязан иметь: **пустое** состояние (онбординг/призыв к действию),
**загрузку** (скелетоны), **ошибку**.

---

## 7. Границы

- Claude Design делает **фронтенд/дизайн**, не backend. Никаких реальных
  запросов к БД или Telegram.
- Данные — моки в форме будущего API.
- Не изобретать auth-флоу: на первом этапе достаточно простого выбора роли /
  входа по ссылке (реальная авторизация ученика — через Telegram-хендофф,
  делается в Claude Code).
- Адаптив обязателен: экран решения и dashboard должны быть удобны на телефоне.
